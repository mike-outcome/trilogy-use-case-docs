Resident Intelligence App — screen mockups

Six interactive screens for the Riverset resident-data build. Each is self-contained and reflects the schema, coverage and extraction limits established in the implementation plan.

Turn on Spec mode (top right of any screen) to reveal the data binding behind every component — source entity, field, document and measured coverage.

Data in these mockups. Aggregate figures — funnel counts, employer mix, income and origin distributions — are real, measured from the Riverset archive. All person-level detail is synthetic. Names, emails, addresses and identifiers are fabricated; the live archive contains full DOBs, licence numbers and partial SSNs that must never reach a public surface.

Design system

Chrome tokens follow the existing Trilogy × Outcome doc set. Chart colour is a separately validated palette — every ramp below passed the contrast, chroma, colour-vision and lightness-band checks against both the light and dark surfaces.

Fixed order, never cycled. Max three on one plot.

Status colours always ship with an icon or a label — colour never carries the meaning alone. Grey is reserved for “no data” and is not a series colour.

Rules the build must hold

These are not stylistic preferences — each one exists because breaking it produces a chart that misleads.

One axis, always.

Never two y-scales on one plot. Two measures of different scale become two charts, small multiples, or both indexed to a common base.

Colour follows the entity, not its rank.

Filtering a series out must not repaint the survivors. Segment A stays blue whether or not it is the larger cohort.

One filter row, above everything it scopes.

Never a filter inside a chart card. Every panel re-renders against the same slice.

Every chart has a table view.

The WCAG-clean twin. Tooltips enhance, never gate — no value is reachable only by hover, and keyboard focus shows what hover shows.

Label selectively.

A number on every point is chaos. Label the extreme, the endpoint, or the one series the story is about; the axis, legend and tooltip carry the rest.

Coverage travels with the number.

A distribution over 54% of the population is a different claim from one over 100%. Say which, on the chart, every time.

Blank is a state, not a zero.

“Not captured” must render as itself. 19% of folders have no usable document and several fields are absent by design — an empty cell that reads as zero is a bug.

Marks stay thin, chrome stays quiet.

Bars ≤24px with a 4px rounded data-end, 2px lines, hairline solid gridlines one step off the surface. The data is the only thing allowed to be loud.

Notes for the build

Shell & tokens

assets/app.css holds every token, component and both theme scopes. assets/app.js renders the nav, wires theme, spec mode, tooltips and table toggles. No dependencies, no build step, no external requests.

Charts are inline SVG

Deliberately no charting library — the geometry is legible in the source so the specs (bar cap, rounded end, surface gap, hit-target size) can be read directly off the markup.

Dark mode is selected, not flipped

Dark steps are chosen for the dark surface and validated as their own set, declared under both the media query and the data-theme scope so the toggle wins in both directions.

These are mockups, not components

The markup is for reading and agreeing, not for lifting into production. Numbers are literals; in the real app every one is a query against the schema in section B of the implementation plan.
